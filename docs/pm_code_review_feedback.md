# PM 관점 코드 피드백

작성일: 2026-07-07

범위: 현재 코드 기준으로 프론트 주요 화면, FastAPI 백엔드, 알림/캘린더 MCP, 테스트/문서를 훑고 PM 관점의 수정 필요 지점을 정리했다. 이 문서는 피드백용이며 코드 수정은 포함하지 않는다.

## 한 줄 요약

핵심 기능의 뼈대는 갖춰져 있지만, 지금은 완성되지 않은 기능이 완성 기능처럼 노출되는 부분과 운영 보안 기본값, 스케줄러 운영 방식이 발표와 배포 리스크다.

## 우선 수정 필요

| 우선순위 | 피드백 | 근거 | 수정 방향 |
|---|---|---|---|
| P0 | 장보기 기능을 완성 기능처럼 노출하고 있다. | `app/frontend/pages/home/sections/HeroSection.jsx`, `CoreFeatureSection.jsx`, `SolutionSection.jsx`, `Header.jsx`, `MobileBottomNav.jsx`, `ShoppingList.jsx`에서 장보기/최저가/가격 비교가 핵심 흐름처럼 보인다. | 장보기 개발이 완료되기 전까지 홈/네비게이션에서는 `준비 중`, `Beta`, 또는 숨김 처리. 발표에서는 "확장 예정 기능"으로 분리. |
| P0 | 개발용 로그인 API가 운영에서도 노출될 수 있다. | `app/backend/api/auth/auth_api.py`의 `/auth/dev-login`은 개발자용 치트키인데 라우터에 항상 포함된다. | `DEV_MODE`일 때만 라우터 등록하거나, 운영 빌드에서는 제거. |
| P0 | 운영 기본값이 개발 모드에 가깝다. | `app/backend/core/config.py`에서 `DEV_MODE=True`, `JWT_SECRET_KEY=YOUR_JWT_SECRET_KEY_HERE`가 기본값이다. `app/backend/main.py`는 CORS `*`, docs `/docs`를 항상 연다. | 운영 환경에서는 기본 시크릿 사용 시 서버 시작 실패, CORS 허용 도메인 제한, Swagger 공개 여부 결정. |
| P0 | 매일 오전 7시 캘린더 잡이 서버 프로세스마다 실행될 수 있다. | `app/backend/main.py` startup에서 `daily_calendar_loop()`를 바로 생성한다. | 배포 시 worker가 여러 개면 중복 등록 위험이 있으므로 단일 스케줄러, 별도 cron, 또는 env flag로 실행 위치를 고정. |

## 다음 단계에서 정리하면 좋은 부분

| 우선순위 | 피드백 | 근거 | 수정 방향 |
|---|---|---|---|

| P1 | 홈 리뷰 영역은 실제 리뷰인지 예시인지 구분이 필요하다. | `ReviewSection.jsx`는 사용자 후기처럼 보이는 문구를 렌더링한다. 현재는 출처/실제성 확인이 어렵고 문자열도 깨져 있다. | 실제 리뷰가 아니면 "사용자 시나리오 예시"로 표현. 실제 리뷰라면 출처와 수집 근거 정리. |

## 기능별 PM 상태

| 기능 | 현재 상태 | PM 판단 |
|---|---|---|
| 소셜 로그인 / JWT | OAuth와 JWT 발급 구조가 있다. | 운영 전 `/dev-login`, 기본 시크릿, 토큰 정책만 정리하면 발표 가능. |
| 냉장고 재료 관리 | CRUD, 보관 기준, 소비 임박 계산 흐름이 있다. | 핵심 기능으로 보여줘도 된다. |
| 영수증 OCR | 업로드, OCR, 후보 확인, 냉장고 입고 흐름이 있다. | 보안/파일 검증 트러블슈팅 소재로 좋다. |
| 레시피 추천 | 검색/추천/상세/저장 흐름이 있다. | 냉장고 데이터와 연결된 핵심 기능으로 설명 가능. |
| 식재료 가이드 / Neo4j | 가이드 그래프와 조회 구조가 있다. | 기술 차별점으로 사용 가능. |
| 알림 / Google Calendar / MCP | 오전 7시 동기화, Google Calendar 연동, RunPod Serverless 경로가 있다. | 발표 가능하되 운영 중복 실행과 실패 표기 정책은 보완 필요. |
| 챗봇 / Agent | 알람 agent와 라우팅/테스트가 있다. | "멀티 에이전트 확장 중"으로 표현하는 편이 안전하다. |
| 장보기 | API, 화면, DB 모델은 있으나 개발 완료 상태로 보기 어렵다. | 핵심 기능이 아니라 후속 확장/Beta로 분리하는 게 안전하다. |

## 발표용으로 바로 쓸 수 있는 정리

- 사용자 문제: 식재료 구매 후 보관, 소비 기한, 활용 메뉴, 알림 관리가 흩어져 있어 낭비가 생긴다.
- 현재 강점: 영수증 OCR 입고, 냉장고 재료 관리, 레시피 추천, 식재료 가이드, 캘린더 알림이 하나의 흐름으로 이어진다.
- 기술 차별점: PostgreSQL은 사용자/재료/이력 관리, Neo4j는 식재료 가이드와 관계 조회, MCP/RunPod는 외부 캘린더 도구 호출을 분리한다.
- 보완할 점: 장보기는 완성 전까지 과장 노출을 줄이고, 운영 보안 기본값과 스케줄러 실행 방식을 먼저 정리해야 한다.

## 추천 작업 순서

1. 장보기 노출을 `Beta` 또는 숨김으로 정리.
2. `/auth/dev-login`, CORS, JWT 기본값, Swagger 공개 여부 운영 정책 반영.
3. 캘린더 오전 7시 잡의 중복 실행 방지 방식 결정.
4. 테스트 문서와 발표 자료를 현재 테스트 파일 기준으로 다시 생성.
5. 발표 자료 변환 전 한글 렌더링을 IDE/브라우저/PDF 기준으로 확인.
