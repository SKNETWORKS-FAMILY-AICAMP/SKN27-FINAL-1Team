# 식재료 가이드 작업 회고

## 1. 원재료명 정규화 기준 설계

**고민:** 만개의레시피 식재료명, 식품DB 영양식품명, 기존 원재료명이 서로 달라 검색·추천·가이드 연결 기준이 불안정했다.

**해결:** 원재료명 기준을 `식품DB 영양식품명 → 서비스용 대표 원재료명 → 기존명 Alias 보존` 순서로 정리했다.

**포인트:** 검색, 추천, 가이드 조회가 같은 원재료 기준으로 동작하도록 데이터 정합성을 확보했다.

## 2. 식품 분류 체계 재설계

**고민:** 전국통합 원재료성 식품 DB는 총 3,704건이지만 `typeNm`이 1개라 실제 서비스 분류 기준으로 부족했다. 반면 식약처 영양성분 DB는 총 302,628건이고 대분류 64개, 중분류 1,271개로 너무 세분화되어 음식·가공식품·원재료가 섞여 있었다.

**해결:** 전국통합 원재료성 식품 DB를 1순위 기준으로 삼고, 중분류 성격의 `foodLv3Nm` 17개를 중심으로 서비스용 분류를 재구성했다. 부족한 항목은 식약처 영양성분 DB로 보완했다.

**포인트:** 공공데이터 분류를 그대로 사용하지 않고, Neo4j에 적재된 실제 기준에 맞춰 4개 대분류·25개 중분류로 정리했다.

## 3. 유사 식재료 통합·분리 기준 수립

**고민:** 생크림/휘핑크림, 가염버터/무염버터, 모짜렐라치즈/슈레드피자치즈처럼 명칭은 유사하지만 영양DB 기준과 실제 사용 맥락이 다른 항목이 있었다.

**해결:** 영양식품명, 사용자 검색 편의성, 서비스 사용 맥락을 기준으로 통합·분리 여부를 개별 검수했다.

**포인트:** 단순 문자열 매칭이 아니라, 영양DB 기준과 서비스 사용성을 함께 고려해 데이터 품질을 관리했다.

## 4. Neo4j 그래프 적재 구조 안정화

**고민:** 원재료, Alias, 가이드, 영양성분, 제철시기, 출처가 연결되어 있어 CSV 수정 시 관계 정합성이 깨질 수 있었다.

**해결:** 노드 CSV 8개, 관계 CSV 8개 구조를 유지하고, 기존 파일명은 유지한 채 내용만 교체하는 방식으로 재적재했다. 이후 `validate_neo4j`로 검증을 반복했다.

**포인트:** 그래프DB 기반 식재료 지식 구조를 안정적으로 관리할 수 있는 적재·검증 흐름을 구축했다.

## 앞으로 작업할 내용

### 1. 식재료 가이드 Agent화

식재료명을 입력하면 보관법, 손질법, 세척법, 신선도 확인법, 제철시기, 영양성분을 Agent가 조회·요약하도록 분리한다.

### 2. Multi-Agent 챗봇 연동

식재료 가이드 Agent를 냉장고 Agent, 레시피 Agent와 연결해 사용자 질문에 따라 필요한 Agent가 분기 처리되도록 구성한다.

### 3. Supervisor Agent 라우팅 설계

가이드 조회, 재고 확인, 레시피 추천, 알림 등록 요청을 한 챗봇에서 처리할 수 있도록 Supervisor가 전체 대화 흐름을 조율한다.

### 4. 공통 응답 JSON 포맷 적용

Guide Agent 응답을 `ok`, `agent`, `action`, `intent`, `message`, `data`, `error`, `requires_confirmation`, `ui`, `meta` 형식으로 통일한다.

### 5. 모호한 식재료 질문 처리

Alias와 대표 원재료명 기준으로 `크림`, `버터`처럼 모호한 질문을 확인 처리해 잘못된 원재료 매칭을 방지한다.
