## 리뷰 발견사항

### P1. Dockerfile의 LightFM 버전 표현

- 위치: `app/backend/Dockerfile:8`
- shell form의 `lightfm>=1.17`에서 `>`가 redirection으로 해석될 수 있다.
- 버전 제약 없이 `lightfm`을 설치하고 출력은 `=1.17` 파일로 향할 수 있어 build 재현성이 깨진다.
- `app/backend/requirements.txt`에 `lightfm>=1.17`을 넣고 Dockerfile은 `pip install -r requirements.txt`만 실행한다.

### P2. 내부 함수 re-export가 경계를 흐림

- 위치: `ai/agents/recipe_agent/recipe_agent.py:39`
- 테스트와 하위 호환을 위해 planner/tool 내부 함수를 re-export한다.
- 기존 호출처를 새 모듈로 이동한 뒤 compatibility layer의 제거 시점을 정한다.

### P2. 실행·품질 단계의 직접 테스트가 부족함

- planner shadow test는 있지만 `execute_plan()`과 `review_recipe_quality()`를 직접 검증하는 단위 테스트는 검색되지 않았다.
- 성공, 빈 결과, tool 오류, fallback 경로만 작은 테스트로 추가한다.

### P2. planner fallback 오류가 관측되지 않음

- 위치: `ai/agents/recipe_agent/recipe_planner.py:178-187`
- 예외가 나면 규칙 planner로 정상 fallback하지만 원인이 기록되지 않는다.
- warning log와 `planner_source` metric을 남긴다.

### P2. 대형 데이터와 model artifact가 Git에 포함됨

- `storage/raw/recipe/recipe.csv`: 약 15MB
- `storage/raw/recipe/cooking_steps.csv`: 약 11MB
- `app/backend/services/recommendation_service/lightfm_model.pkl`: 약 1.4MB
- 배포에 필요한 artifact만 versioned storage나 Git LFS에서 받도록 검토한다.

### P2. 동일한 가공 데이터가 중복됨

- `recipe_ingredient_alias.csv`, `recipe_fix.csv`가 `ai/recommendation/data`와 `storage/processed/recipe`에 각각 존재한다.
- source of truth를 하나로 정하고 생성 script와 checksum으로 재현한다.

## 주석 개선

- LLM planner가 규칙 planner로 fallback하는 조건
- tool 오류가 전체 응답 실패인지 빈 결과인지 판단하는 정책
- quality review note가 최종 응답에 미치는 영향
- model과 데이터 artifact의 버전 호환 관계

함수명과 dataclass가 이미 명확한 곳에는 설명 주석을 추가하지 않아도 된다.
